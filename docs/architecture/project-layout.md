# Project layout

File-by-file walk of the source tree. Skim this if you're contributing
or trying to find the code behind a specific feature.

## Top-level

```text
.
├── cmd/macontrol/                  # binary entry point + CLI subcommands
├── internal/                       # everything that compiles into the binary
│   ├── capability/                 # macOS version detection + feature gates
│   ├── config/                     # env loader
│   ├── domain/                     # 11 macOS-control packages (no Telegram)
│   ├── runner/                     # subprocess wrapper + Fake for tests
│   ├── telegram/                   # bot, callbacks, keyboards, flows, handlers
│   └── version/                    # link-time version metadata
├── docs/                           # this directory
├── launchd/                        # LaunchAgent plist template
├── scripts/                        # install.sh
├── sudoers.d/                      # sudoers template
├── .github/                        # CI, issue templates, dependabot
├── Makefile                        # build, test, lint, etc.
├── .goreleaser.yaml                # release pipeline
├── release-please-config.json      # version bump rules
├── version.txt                     # current version (release-please-managed)
├── go.mod / go.sum                 # Go module
├── README.md                       # landing page
├── CHANGELOG.md                    # auto-generated by release-please
└── (other community files)
```

## `cmd/macontrol/`

The CLI binary. Each file is one subcommand:

```text
cmd/macontrol/
├── main.go         # argv dispatch: run, setup, service, doctor, version
├── daemon.go       # `run` subcommand: wire deps, start the bot
├── setup.go        # `setup` subcommand: interactive first-run wizard
├── service.go      # `service install/uninstall/start/stop/status/logs`
├── doctor.go       # `doctor` subcommand: capability + brew + sudoers report
├── assets.go       # plistBody() and sudoersBody() generators
└── user.go         # currentUser() — wraps user.Current() with a fallback
```

Why so much code in `cmd/`: macontrol is a CLI tool with multiple
subcommands, not just a daemon. The setup wizard alone is ~200 lines
because it's truly interactive.

## `internal/runner/`

The single subprocess shim every domain package uses:

```text
internal/runner/
├── runner.go        # Runner interface, Exec implementation, *Error type
├── fake.go          # in-memory Fake for tests (rule-based dispatch)
├── runner_test.go   # exec, error capture, timeout
└── extras_test.go   # Sudo behavior, Fake concurrency
```

Every macOS interaction goes through this. `runner.Exec` wraps
`exec.CommandContext`; `runner.Sudo` prepends `sudo -n` so missing
sudoers entries fail fast.

## `internal/capability/`

macOS version detection and feature-gate computation:

```text
internal/capability/
├── detect.go         # ParseVersion, AtLeast, Detect, DeriveFeatures, Report.Summary
└── detect_test.go
```

Every handler that depends on a version-gated feature checks
`d.Capability.Features.<flag>` before dispatching. The gate values are
computed once at startup.

## `internal/config/`

Typed env loader:

```text
internal/config/
├── config.go        # Load(), DefaultLogPath()
└── config_test.go
```

Reads the bot token and whitelist from the macOS Keychain via
`internal/keychain`. No env vars, no config file, no migration.
Returns a friendly "missing X; run `macontrol setup`" error when
either Keychain entry is absent.

## `internal/version/`

Three vars set by GoReleaser at link time:

```text
internal/version/
├── version.go        # Version, Commit, Date + String()
└── version_test.go
```

Tiny but important — `macontrol --version` reads these.

matches the callback namespace. So if you see `snd:up:5`, you know to
look in `handlers/snd.go` for the dispatch and `keyboards/snd.go` for
the layout.

## `launchd/`, `sudoers.d/`, `scripts/`

Operational artifacts:

```text
launchd/com.amiwrpremium.macontrol.plist     # template; setup wizard rewrites paths
sudoers.d/macontrol.sample                   # template; setup wizard substitutes username
scripts/install.sh                           # manual install bootstrap
```

These ship in the GoReleaser tarball alongside the binary. The setup
wizard prefers to generate fresh content (with the right paths and
username) over copying these templates — but they're useful for
manual installs and as documentation of the canonical contents.

## Where to add new code

- **A new dashboard button on an existing category** → edit
  `internal/telegram/keyboards/<ns>.go` (add the button) +
  `internal/telegram/handlers/<ns>.go` (handle the new callback action).
- **A new flow on an existing category** → add a new file in
  `internal/telegram/flows/<flowname>.go`, then trigger it from the
  handler with `d.FlowReg.Install(chatID, flows.NewFlowname(svc))`.
- **A new category** → see
  [Development → Adding a capability](../development/adding-a-capability.md).
- **A new domain function on an existing package** → add the method to
  `internal/domain/<pkg>/<file>.go` and a test in
  `<pkg>/<pkg>_test.go`.
- **A new env var** → declare a struct field in
  `internal/config/config.go`, document in
  `docs/configuration/env.md`.
